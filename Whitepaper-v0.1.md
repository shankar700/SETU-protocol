## Abstract

Generative UI (GenUI) — the real-time, AI-driven assembly of interactive interfaces — is transitioning from theoretical concept to production reality. However, existing protocols address only the rendering pipeline within a single company's boundary. No open standard exists for the layer above: a trusted, decentralised marketplace where enterprise UI components are registered by providers, discovered by AI agents, and assembled into unified user experiences across organisational boundaries.

This paper introduces **SETU** (System of Extensible & Tangible User Interfaces), an open protocol that defines this missing marketplace layer. SETU enables AI agents to query a decentralised registry of trusted UI components, render competing provider interfaces simultaneously, and route user transactions back to the originating provider — all within a single agent surface, without requiring the user to switch applications.

SETU is designed to be complementary to, not competitive with, existing protocols. It sits above MCP, AG-UI, and A2UI in the agentic stack, acting as the trusted orchestration and discovery layer those protocols currently lack.

The name *Setu* (सेतु) is the Sanskrit and Hindi word for bridge — a fitting metaphor for a protocol that bridges user intent, AI agents, and enterprise provider interfaces into a single, coherent experience.

## 1. Introduction

The Model Context Protocol (MCP), introduced by Anthropic in November 2024, demonstrated that a well-timed, well-designed open standard could achieve near-universal adoption across competing AI vendors within twelve months [1]. Its success validated a broader principle: the agentic AI ecosystem is hungry for open standards that reduce integration friction and expand capability without vendor lock-in.

MCP solved the connection between AI agents and tools and data sources. The Agent-User Interaction Protocol (AG-UI) and Google's Agent-to-UI (A2UI) subsequently addressed the rendering of interfaces within single-app boundaries [2][3]. However, a critical layer remains unaddressed: the cross-company marketplace where enterprise UI components are published by providers, discovered by agents, and rendered across organisational boundaries with verifiable trust guarantees.

This is the layer SETU proposes to fill.

The problem is not theoretical. As Nielsen Norman Group observed in March 2026, simple interactive elements — buttons, form fields, checkboxes — are beginning to appear contextually within AI conversations, generated by AI when it determines they would improve the interaction [4]. But these implementations are entirely within single-company walls. When an AI agent needs to present a ride-booking UI from Uber alongside one from Lyft, rendered simultaneously inside a single agent interface, no protocol exists to facilitate that interaction safely and consistently.

SETU defines that protocol.

### 1.1 Design Philosophy

SETU is guided by four principles:

**Open by default.** The protocol specification is published under Apache 2.0. No single entity — including the protocol's creator — controls the registry or the standard.

**Trust as infrastructure.** Every provider interaction carries a verifiable trust tier. The word "Trusted" is not marketing — it is a technical contract enforced at the protocol layer.

**Extensible by design.** Any company, from a one-person startup to a Fortune 500 enterprise, can register a UI component in the SETU registry. The protocol is the floor, not the ceiling.

**Tangible over conversational.** SETU exists because chat is not the right interface for every user need. Booking a flight, ordering food, or tracking a delivery are fundamentally visual, interactive tasks. SETU makes them tangible within the agent surface.

## 2. Problem Statement

### 2.1 The Fragmentation of the Agentic User Experience

Today's AI agent interactions suffer from a fundamental disconnect: agents have become capable of understanding complex user intent, but they remain unable to present the right interactive interface at the moment of intent. The result is a degraded user experience in which:

- The agent understands the user wants to book a ride, but returns a text response with a URL
- The user must leave the agent surface, open a separate application, re-authenticate, and complete the task manually
- The agent receives no feedback on whether the transaction was completed
- The user's preference history across providers remains siloed in separate applications

This is not a capability failure — it is an infrastructure failure. The tools to render rich, interactive UI from AI agents exist (AG-UI, A2UI, MCP Apps). What does not exist is the trusted marketplace layer that connects those rendering tools to a curated ecosystem of provider UI components.

### 2.2 The Single-Company Boundary Problem

Every existing GenUI protocol operates within a single company's trust boundary. AG-UI connects an agent backend to a frontend application owned by the same developer [2]. A2UI enables an agent to describe UI that a host application renders using its own component library [3]. MCP Apps allow MCP servers to return interactive interfaces within a sandboxed iframe [5].

These are valuable — but they solve a different problem. None of them answer the question: *how does an AI agent safely render a UI component from Uber inside an interface owned by a different company, with verifiable trust, consistent styling conventions, and a clear data boundary?*

That is the SETU problem.

### 2.3 The Trust Gap

Rendering third-party UI inside an agent surface introduces security and trust challenges that existing protocols explicitly avoid. A2UI's design philosophy is "safe like data, but expressive like code" — it sends JSON blueprints rather than executable HTML, precisely because executing arbitrary third-party code is dangerous [3].

SETU extends this safety model to the marketplace layer. By requiring providers to publish cryptographically signed manifests, undergo identity verification, and comply with a tiered trust framework, SETU ensures that the agent's user can trust the UI bricks being assembled on their behalf.

## 3. Related Work

### 3.1 Model Context Protocol (MCP)

MCP, introduced by Anthropic in November 2024, standardises how AI agents connect to external tools, APIs, and data sources [1]. It defines a client-server architecture in which MCP servers expose tools and resources, and MCP clients (AI agents) invoke them. MCP achieved cross-vendor adoption within twelve months, with support from Anthropic, OpenAI, Google DeepMind, and Microsoft.

MCP's success is the direct inspiration for SETU's open-protocol-first approach. SETU adopts MCP's lessons: ship open, build the ecosystem before building the product, and donate governance to a neutral foundation.

SETU is complementary to MCP. Where MCP connects agents to tools and data, SETU connects agents to trusted UI components.

### 3.2 AG-UI Protocol

AG-UI, introduced by CopilotKit in May 2025, standardises real-time, bidirectional communication between agent backends and user-facing frontend applications [2]. It defines a stream of typed events — messages, tool calls, state patches, lifecycle signals — that flow between agent and UI.

AG-UI operates within a single deployment boundary. SETU's render layer builds on AG-UI's event model to extend it across organisational boundaries.

### 3.3 Google A2UI

A2UI, open-sourced by Google in December 2025, defines a declarative JSON format for agent-generated user interfaces that render natively across web, mobile, and desktop platforms [3]. Its core design principle — "safe like data, but expressive like code" — separates UI intent (generated by the agent) from UI execution (rendered by the host application).

SETU adopts A2UI as its preferred render wire format. SETU provider bricks can be described in A2UI's component schema, enabling any A2UI-compatible renderer to display SETU components.

### 3.4 MCP-UI and MCP Apps

MCP-UI, pioneered by Monday.com and subsequently incorporated into the MCP standard as MCP Apps, enables MCP servers to return interactive UI components rendered within the agent interface [5]. Its iframe-based sandboxing approach provides security at the cost of visual integration — components rendered in iframes are visually disconnected from the host application.

SETU's render layer adopts A2UI's native-first approach rather than iframe sandboxing, providing better visual integration while maintaining security through the trust framework.

### 3.5 GenUI Research

Academic research on generative UI has formalised the design space. A 2025 ACM DIS study proposed a working definition of GenUI as an interface created, refined, or evolved through computational generative processes, identifying five constitutive elements: computational co-creation, expanded design space exploration, representation fluidity, contextual adaptation, and synthesis over selection [6].

Google Research demonstrated in 2025 that human raters strongly prefer generative UI interfaces over standard LLM text outputs, validating the commercial and experiential case for the paradigm [7].

The same ACM DIS conference surfaced a critical design constraint from industry practitioners: *"Chat window is not the answer to everything... not everything is conversational"* [8]. SETU is the infrastructure response to this constraint.

## 4. The SETU Protocol

### 4.1 Definition

SETU is an open protocol that defines:

1. A **manifest schema** for providers to register trusted UI components in a decentralised registry
2. A **query interface** for AI agents to discover components by capability, geography, and trust tier
3. A **render protocol** for transmitting component descriptions from the registry to the agent's rendering layer
4. A **transaction protocol** for routing user actions back to provider backends and logging revenue events
5. A **trust framework** for verifying provider identity and enforcing security standards

### 4.2 Positioning in the Agentic Stack

```
┌──────────────────────────────────────────────────┐
│                   USER INTENT                    │
│         "Book me a ride to the airport"          │
└────────────────────┬─────────────────────────────┘
                     │
┌────────────────────▼─────────────────────────────┐
│               AI AGENT LAYER                     │
│     (Claude, GPT, Gemini, open-source agents)    │
└────────────────────┬─────────────────────────────┘
                     │  queries
┌────────────────────▼─────────────────────────────┐
│            SETU REGISTRY LAYER  ◄── THIS PAPER   │
│   Decentralised provider marketplace             │
│   Trust scoring · Capability discovery           │
└──────┬─────────────────────┬────────────────────┘
       │ returns manifests   │
┌──────▼──────────┐   ┌──────▼──────────┐
│  Provider A     │   │  Provider B     │
│  (Uber brick)   │   │  (Lyft brick)   │
└──────┬──────────┘   └──────┬──────────┘
       │ A2UI / AG-UI        │
┌──────▼──────────────────────▼──────────────────┐
│              RENDER LAYER                       │
│   A2UI / AG-UI / MCP Apps wire formats         │
└──────────────────────────────────────────────┘
```

SETU does not replace any layer in this stack. It is the orchestration and discovery layer that coordinates across all of them.

### 4.3 Core Protocol Primitives

SETU defines four core primitives:

**SETU.discover(intent)** — Query the registry for providers matching a capability and context. Returns a ranked list of provider manifests sorted by trust score and relevance.

**SETU.render(manifest, context)** — Request a UI component from a provider, passing the user's session context. Returns an A2UI or AG-UI compatible component description.

**SETU.transact(component_id, action, payload)** — Route a user action to the provider backend. Returns a transaction receipt.

**SETU.verify(provider_id)** — Query the trust framework for a provider's current trust tier, compliance status, and SLA record.

## 5. Architecture

SETU is a four-layer protocol stack. Each layer has a clear responsibility and a clear interface to the layers above and below.

### Layer 1 — Intent Layer

The intent layer parses user intent into a structured SETU query. This is typically handled by the AI agent, which extracts:

```json
{
  "capability": "ride_booking",
  "context": {
    "destination": "airport",
    "time": "2026-03-16T18:00:00+05:30",
    "user_location": { "lat": 17.4401, "lng": 78.3489 }
  },
  "trust_tier_minimum": "registered",
  "geography": "IN",
  "max_results": 5
}
```

### Layer 2 — Registry Layer

The registry layer is the core of SETU. It is a decentralised, content-addressed store of provider manifests. Each manifest describes a UI component's capabilities, rendering endpoint, trust tier, transaction fee, and data schema.

The registry exposes a discovery API:

```
GET /discover?capability=ride_booking&geography=IN&trust_tier=verified
```

Returns an array of provider manifests, ranked by trust score.

The registry is designed to be decentralised over time — beginning with a canonical reference implementation hosted by the SETU foundation, and migrating toward a distributed architecture governed by the community.

### Layer 3 — Render Layer

The render layer handles the transmission of UI component descriptions from provider endpoints to the agent's rendering surface. SETU adopts A2UI as its preferred wire format for this layer.

Providers expose a SETU-compliant render endpoint:

```
POST /setu/render
{
  "session_context": { ... },
  "user_preferences": { ... },
  "render_format": "a2ui_v0.8"
}
```

The provider responds with an A2UI component tree that the agent's renderer displays natively.

### Layer 4 — Transaction Layer

The transaction layer routes user actions from the rendered component back to the provider backend. It is responsible for:

- Authenticating the user with the provider (using OAuth 2.0 or provider-specific flows)
- Routing the action payload to the provider's transaction endpoint
- Logging the revenue event for billing purposes
- Returning a signed transaction receipt to the agent

## 6. Provider Manifest Schema

Every provider registered in SETU publishes a manifest. The manifest is a JSON document that describes the provider's UI component, capabilities, and trust credentials.

```json
{
  "$schema": "https://setu-protocol.dev/schemas/manifest/v0.1.json",
  "provider_id": "did:setu:uber-technologies-inc",
  "name": "Uber Ride Booking",
  "description": "Real-time ride booking, fare estimation, and live tracking",
  "version": "1.0.0",
  "capabilities": [
    "ride_booking",
    "ride_tracking",
    "fare_estimate",
    "ride_cancel"
  ],
  "ui_endpoint": "https://setu.uber.com/components/ride-booking",
  "render_formats": ["a2ui_v0.8", "react_component", "web_component"],
  "geographies": ["IN", "US", "GB", "AU", "SG", "AE"],
  "trust_tier": "verified",
  "trust_certificate": "https://setu-protocol.dev/trust/certs/uber-2026-03",
  "transaction_fee": 0.02,
  "sla": {
    "availability": 0.999,
    "response_time_ms": 200
  },
  "data_schema": {
    "requires": ["user_location", "destination"],
    "optional": ["payment_method_preference", "ride_type_preference"],
    "retention_policy": "session_only",
    "data_residency": ["IN", "US"]
  },
  "branding": {
    "primary_color": "#000000",
    "logo_url": "https://setu.uber.com/assets/logo.svg"
  },
  "published_at": "2026-03-01T00:00:00Z",
  "signature": "eyJhbGciOiJFZERTQSJ9..."
}
```

The manifest is cryptographically signed by the provider and verified by the SETU registry on publication. Any change to the manifest invalidates the signature and requires re-verification.

## 7. Trust Framework

Trust is SETU's central differentiator. Every component in the registry carries a verifiable trust tier, and agents can filter by trust tier based on the sensitivity of the user's request.

### 7.1 Trust Tiers

| Tier | Requirements | Use Case |
|------|-------------|----------|
| **Verified** | Legal entity verified · SOC2 Type II · SLA ≥ 99.9% · Security audit · Data residency declaration | Financial transactions, health data, identity |
| **Registered** | Legal entity verified · Basic compliance · SLA ≥ 99% | E-commerce, bookings, subscriptions |
| **Community** | Open source · No commercial transactions · Sandbox only | Developer tools, demos, experiments |

### 7.2 Trust Scoring

Beyond the tier classification, SETU maintains a continuous trust score for each provider based on:

- **Uptime compliance** — actual SLA performance vs. declared SLA
- **Incident history** — security incidents, data breaches, user complaints
- **Community rating** — developer and user feedback
- **Compliance currency** — whether certifications are current and valid

Trust scores are public, versioned, and auditable. Any agent querying the registry can retrieve the full trust history of any provider.

### 7.3 Trust Revocation

If a provider's trust tier is downgraded or revoked, the SETU registry broadcasts a revocation event to all subscribed agents. Agents are expected to immediately stop rendering components from revoked providers and surface appropriate messaging to users.

## 8. End-to-End Workflow

The following illustrates a complete SETU interaction for a ride-booking request.

### Step 1 — User Intent
```
User → Agent: "Book me a ride to Hyderabad Airport at 6pm tomorrow"
```

### Step 2 — Intent Parsing
The agent extracts structured intent and constructs a SETU discovery query.

### Step 3 — Registry Discovery
```
GET /discover?capability=ride_booking&geography=IN&trust_tier=registered
```

Registry returns: Uber (verified, score: 98), Ola (verified, score: 96), Rapido (registered, score: 87)

### Step 4 — Parallel Component Render
The agent requests UI components from all three providers simultaneously:

```
POST https://setu.uber.com/setu/render      → Uber ride UI
POST https://setu.olacabs.com/setu/render   → Ola ride UI  
POST https://setu.rapido.bike/setu/render   → Rapido ride UI
```

### Step 5 — Simultaneous Display
All three provider UIs are rendered side-by-side inside the agent interface. The user sees live pricing, availability, and ETA from all three providers in one view.

### Step 6 — User Selection and Transaction
The user taps "Book" on Uber's component. The SETU transaction layer routes the action:

```
POST /setu/transact
{
  "provider_id": "did:setu:uber-technologies-inc",
  "component_id": "ride-booking-v1",
  "action": "confirm_booking",
  "payload": { "ride_id": "abc123", "payment_method": "upi" }
}
```

### Step 7 — Confirmation
Uber's component updates inline to show booking confirmation, driver name, vehicle details, and live ETA. The user never left the agent interface.

### Step 8 — Revenue Event
The transaction layer logs the event. Uber is billed a 2% transaction fee. The SETU protocol treasury receives the fee and distributes it according to the governance model.

## 9. Feasibility Analysis

### 9.1 Technical Feasibility

The core technical components required for SETU are mature:

- **Decentralised registries** — established patterns exist in npm, PyPI, and emerging Web3 identity systems
- **Declarative UI rendering** — A2UI (Google, 2025) and AG-UI (CopilotKit, 2025) provide production-ready wire formats
- **Cryptographic manifest signing** — standard practice in package registries and DNS security
- **Sandboxed component execution** — A2UI's declarative approach eliminates arbitrary code execution risks

The primary technical challenge is **latency**. Rendering 3–5 provider UIs simultaneously requires sub-200ms response times from provider render endpoints. SETU's SLA requirements for the Verified tier (≥99.9% availability, ≤200ms response) are designed to enforce this.

### 9.2 Adoption Feasibility

MCP's adoption trajectory provides the most relevant precedent. MCP grew from zero to 97 million monthly SDK downloads within twelve months [1], driven by three factors: a clear developer value proposition, an open governance model, and early adoption by a small number of high-credibility anchor integrators.

SETU's adoption strategy mirrors this:

1. **Publish the open spec first** — before any commercial product
2. **Build the auto-adapter layer** — enabling providers to register without rewriting their existing APIs
3. **Anchor with 3–5 credible providers** — one Uber-equivalent integration makes the protocol real
4. **Donate governance early** — preventing the "one company controls this" concern that kills enterprise adoption

### 9.3 Risks

**Risk 1: Big tech OS-layer encroachment.** Apple Intelligence and Google Gemini are building agent-driven UI capabilities into their operating systems. The mitigation is to win the standard before they ship — an open protocol donated to a neutral foundation is significantly harder to displace than a proprietary implementation, as the MCP story demonstrated.

**Risk 2: Provider cold-start problem.** Providers have no incentive to build SETU components if there are no agents using SETU, and agents have no incentive to query SETU if there are no providers. The auto-adapter layer — which converts existing REST APIs into SETU-compliant bricks automatically — is the key mitigation. Zero-friction registration dramatically lowers the cold-start barrier.

**Risk 3: Trust framework gaming.** Providers may misrepresent their compliance posture to achieve higher trust tiers. SETU mitigates this through continuous monitoring, community reporting mechanisms, and mandatory third-party audits for the Verified tier.

## 10. Monetisation Model

SETU as a protocol is free and open. SETU as infrastructure is sustainable through three revenue streams:

### 10.1 Provider Listing Fees

Providers pay a monthly or annual fee to be discoverable in the canonical SETU registry. Fees are tiered by the number of registered components, geographies, and trust tier. Community tier registration is free.

This provides predictable, provider-funded revenue that is independent of consumer adoption velocity.

### 10.2 Transaction Revenue Share

A 2–5% fee on all transactions completed through SETU-rendered UI components. This fee is declared in the provider manifest and is non-negotiable for Verified and Registered tier providers.

This aligns SETU's incentives with provider success: SETU earns when providers earn.

### 10.3 Infrastructure Services

Even if the protocol specification is adopted by competing registry operators, running the canonical, authoritative registry — with trust scoring, compliance monitoring, uptime guarantees, and SLA enforcement — is a paid infrastructure service.

This mirrors the relationship between the HTTP standard (free and open) and cloud hosting providers (paid infrastructure on top of the standard).

## 11. Governance

### 11.1 Principles

SETU's governance is designed to prevent any single entity — including the protocol's creator — from controlling the standard or the registry.

1. **The specification is open** — changes require community consensus through a public RFC process
2. **The registry is auditable** — all provider manifests, trust scores, and revocation events are publicly queryable
3. **The foundation is neutral** — SETU will seek governance under an established open-source foundation (Linux Foundation, Apache Software Foundation, or equivalent)

### 11.2 RFC Process

Changes to the SETU specification follow a four-stage RFC process:

1. **Proposal** — any community member may open a GitHub issue proposing a change
2. **Discussion** — 30-day open comment period
3. **Draft RFC** — a formal specification document is written and reviewed
4. **Acceptance** — accepted by a governance committee of community-elected representatives

### 11.3 Roadmap to Neutral Governance

The protocol is currently governed by its creator. The roadmap to neutral governance is:

- **v0.1** — Creator-governed, open specification, Apache 2.0
- **v0.5** — Community steering committee formed, RFC process active
- **v1.0** — Governance donated to neutral foundation

## 12. Open Questions

The following questions require community input before SETU v1.0 is finalised:

### Technical
- What is the canonical identity system for provider DIDs — W3C DID standard, or a SETU-specific scheme?
- How should SETU handle real-time bidding between providers for a given intent query?
- What is the right sandboxing model for providers that cannot adopt A2UI — should iframes be supported as a fallback?
- How are versioned components handled — when a provider ships a new component version, what is the migration path?

### Governance
- Should the registry be fully decentralised from v1.0, or begin centralised and migrate?
- What is the appeals process for providers whose trust tier is downgraded?
- How are conflicts of interest on the governance committee managed?

### Business
- What transaction fee rate maximises provider adoption without undermining SETU's sustainability?
- Should SETU operate a white-label private registry product for enterprises, or focus exclusively on the public registry?

## 13. Conclusion

The agentic AI ecosystem has built powerful tools for connecting agents to data, tools, and other agents. The user interface layer — the tangible, interactive surface where users actually experience AI — remains fragmented, siloed, and untrustworthy across organisational boundaries.

SETU proposes to bridge this gap.

By defining an open, extensible, trust-first protocol for the cross-company UI component marketplace, SETU enables a future in which AI agents can assemble the right interface from the right provider at the right moment — without the user ever leaving their agent surface, and without the developer having to build bespoke integrations for every provider they want to support.

The name *Setu* — Sanskrit for bridge — captures this vision precisely. A bridge does not replace the places it connects. It makes moving between them effortless.

SETU is that bridge.
